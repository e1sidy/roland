# Concepts

Deep dive into Roland's core abstractions and design decisions.

## Workspace Lifecycle

A **workspace** is an ephemeral task directory that exists only while a task is active. It is created during `pickup` and destroyed during `done`.

### Directory naming

Workspaces live under `ROLAND_HOME/tasks/` and are named with a slug derived from the task ID and title:

```
st-a1b2-fix-authentication-bug
```

The slug is generated by:
1. Starting with the task ID (e.g., `st-a1b2`)
2. Appending a hyphenated, lowercased version of the title
3. Replacing non-alphanumeric characters with hyphens
4. Collapsing consecutive hyphens
5. Truncating to 50 characters

### Task ID extraction

Roland can extract the task ID from a slug or path using the pattern `st-[a-z0-9]+(\.[0-9]+)*`. This supports both flat IDs (`st-a1b2`) and hierarchical IDs (`st-a1b2.1.2`).

When a command accepts an optional `[task-id]` argument, Roland detects the task from the current working directory if the argument is omitted. This means you can `cd` into a task directory and run commands without specifying the ID.

### Lifecycle states

```
                    pickup
                      │
            ┌─────────▼─────────┐
            │     WORKSPACE      │
            │  ┌──────────────┐  │
            │  │ Worktrees    │  │
 work ◄────►│  │ CLAUDE.md    │  │◄────► checkpoint
            │  │ Hooks/Skills │  │
            │  └──────────────┘  │
            └─────────┬─────────┘
                      │
               ship   │   done
               (PRs)  │   (cleanup)
                      ▼
                  [REMOVED]
```

The workspace is the agent's working environment. It contains:
- Symlinks to git worktrees (one per repo)
- A generated `CLAUDE.md` with persona instructions and task context
- Hook scripts and settings
- Symlinked skill directories

## Worktree Strategy

Roland uses **centralized git worktrees** with symlinks for isolation.

### Layout

```
ROLAND_HOME/
├── repos/
│   └── backend/           # Bare-ish clone (main working tree)
├── worktrees/
│   └── backend/
│       ├── st-a1b2/       # Worktree for task st-a1b2
│       └── st-c3d4/       # Worktree for task st-c3d4
└── tasks/
    └── st-a1b2-fix-auth/
        └── backend → ../../worktrees/backend/st-a1b2  (symlink)
```

### How it works

1. **Repos are cloned once** into `ROLAND_HOME/repos/`. This is the "main" working tree.
2. **Worktrees are created per-task** under `ROLAND_HOME/worktrees/<repo>/<branch>/`. Each worktree gets a new branch based on the configured base branch.
3. **Symlinks connect tasks to worktrees**. The task directory contains a symlink named after the repo pointing to the worktree.
4. **`.roland-base`** is written inside each worktree recording the PR target branch (e.g., `origin/main`). This is read by `roland ship` to set the PR base.

### Benefits

- **Isolation**: Each task works on its own branch without interfering with others.
- **Speed**: Worktrees share the Git object store, so creation is fast.
- **Cleanup**: `roland done` removes the worktree cleanly via `git worktree remove`.

### Post-setup scripts

Each repo can have a `post_setup` script that runs after worktree creation. This is useful for installing dependencies:

```yaml
repos:
  frontend:
    url: https://github.com/org/frontend.git
    post_setup: ./scripts/setup.sh
```

The script runs with the worktree as its working directory. Failures are non-fatal.

### Custom branch naming

By default, the task ID is used as the branch name. To customize this, set `branch_name` to a script path. The script receives the full task JSON on stdin and should output the branch name to stdout.

## Persona System

Personas are behavior templates that shape how an AI agent approaches a task. They are injected as the first section of the generated `CLAUDE.md`.

### Built-in personas

Roland ships 4 personas embedded in the binary:

| Persona | Focus |
|---------|-------|
| `builder` | Implementation — writes code, tests, commits |
| `researcher` | Investigation — explores codebases, documents findings |
| `reviewer` | Quality — reviews code, identifies issues, suggests fixes |
| `planner` | Strategy — breaks work into tasks, creates plans |

### Custom personas

Custom personas are Markdown files stored in `ROLAND_HOME/personas/`:

```
ROLAND_HOME/personas/
├── devops.md          # Pure custom persona
└── builder.md         # Override of built-in
```

**Precedence**: Custom personas with the same name as a built-in take precedence. Deleting a custom override reverts to the built-in version.

### Creating custom personas

```bash
# Empty template
roland persona create devops

# Copy from existing
roland persona create devops --from builder

# Edit in your $EDITOR
roland persona edit devops
```

### How personas are used

During `roland pickup`, the persona content is prepended to the task-specific `CLAUDE.md`:

```markdown
# Builder                        ← Persona content
...persona instructions...

---

# Task Context                   ← Auto-generated
- **Task ID**: st-a1b2
- **Title**: Fix authentication bug
- **Priority**: P2
...

---

# Roland                         ← Embedded base instructions
...roland workspace rules...
```

## Hook System

Hooks inject context into AI agent sessions. They use a 4-layer architecture.

### Layer 1: Hook Definitions

Each hook has:
- **Name** — unique identifier (e.g., `slate-instructions`)
- **Source** — where it is installed: `home` (global) or `task` (per-task)
- **Event** — agent event trigger (e.g., `SessionStart`, `PreCompact`)
- **Matcher** — tool filter (`*` for all tools, or a specific tool name)
- **Timeout** — seconds before the hook is killed (default: 10)
- **Content generators** — functions that produce scripts for each agent type

### Layer 2: Registry

The registry is a collection of hooks, queryable by name or source. The default registry comes pre-loaded with 5 built-in hooks:

| Hook | Source | Trigger | Purpose |
|------|--------|---------|---------|
| `slate-instructions` | home | SessionStart | Slate CLI cheatsheet |
| `slate-ready-tasks` | home | SessionStart | Dynamic: runs `slate ready` |
| `roland-instructions` | home | SessionStart | Roland CLI cheatsheet |
| `roland-repos` | home | SessionStart | Dynamic: runs `roland repo list` |
| `roland-task-context` | task | SessionStart | Dynamic: shows task + latest checkpoint |

### Layer 3: Manager

The manager orchestrates installation:
- **Install** — writes a hook script and wires it into agent settings
- **Uninstall** — removes the script and settings entry
- **Sync** — ensures exactly the enabled hooks are installed (idempotent)

### Layer 4: Delivery

Delivery is agent-specific:

**Claude Code**: Each hook becomes a bash script in `hooks/<name>.sh`, referenced from `.claude/settings.json`:

```json
{
  "hooks": [
    {
      "type": "SessionStart",
      "command": "bash /path/to/hooks/slate-instructions.sh"
    }
  ]
}
```

**OpenCode**: All enabled hooks are combined into a single JS plugin at `.opencode/plugins/roland-hooks.js`.

### Enabling and disabling

Hooks are enabled by default. Disable them via config:

```bash
roland config hooks disable slate-ready-tasks
roland config hooks sync
```

Or use the hook commands:

```bash
roland hook remove slate-ready-tasks   # Disable + uninstall
roland hook add slate-ready-tasks      # Enable + install
```

## Skill System

Skills are reusable context directories that provide specialized knowledge to agents. They are auto-injected into task workspaces based on matching criteria.

### Skill structure

A skill is a directory containing a `SKILL.md` file (and any other files the agent needs):

```
my-skill/
├── SKILL.md           # Required: skill description and instructions
├── examples/
│   └── usage.go
└── templates/
    └── config.yaml
```

### Registration

```bash
# Copy into ROLAND_HOME/.skills/
roland skill add ./my-skill --name testing

# Keep at external location (symlinked)
roland skill add /path/to/skill --name testing --external
```

Skills are tracked in `ROLAND_HOME/.skills/skills.json`.

### Matching criteria

Skills can be tagged with matching criteria:

```bash
roland skill tag testing --personas builder,researcher --types bug,feature --tags backend
```

### OR matching logic

During `roland pickup`, skills are matched against the task context using OR logic across three dimensions:

1. **Personas**: If the skill has personas defined and the active persona matches any of them, the skill is injected.
2. **Task types**: If the skill has task types defined and the task's type matches any of them, the skill is injected.
3. **Tags/labels**: If the skill has tags defined and any of the task's labels match any of the tags, the skill is injected.

If a match is found in **any** dimension, the skill is injected. If all three dimensions are empty, the skill is never auto-injected (manual-only).

### Injection mechanism

Skills are injected by creating a symlink in the task's `.claude/skills/` directory:

```
tasks/st-a1b2-fix-auth/
└── .claude/
    └── skills/
        └── testing → /path/to/skill/directory
```

### Manual injection/ejection

```bash
roland skill inject testing st-a1b2    # Add to workspace
roland skill eject testing st-a1b2     # Remove from workspace
```

## Atomic Pickup with Rollback

The `pickup` command is designed for crash safety. Each step has explicit rollback logic:

| Step | Action | Rollback on failure |
|------|--------|-------------------|
| 1 | Open Slate store | None needed |
| 2 | Get task details | None needed |
| 3 | Claim task | None needed (nothing happened yet) |
| 4 | Create workspace | Release claim |
| 5 | Set attributes | Remove workspace + release claim |
| 6 | Create worktrees | Remove earlier worktrees + workspace + claim |
| 7 | Write CLAUDE.md | Remove worktrees + workspace + claim |
| 8 | Install hooks/skills | **Warn only** — non-critical |

This ensures that a failed pickup never leaves the system in an inconsistent state. A task is never left claimed without a workspace, and a workspace is never left without a valid claim.

## Session Continuity

Roland supports session continuity through checkpoints and the `work` command.

### Checkpoints

At any point during work, agents (or humans) can record structured progress:

```bash
roland checkpoint --done "Implemented auth middleware" \
  --next "Add unit tests" \
  --blockers "Need API key" \
  --decisions "Using JWT instead of sessions" \
  --files "auth.go,auth_test.go"
```

Checkpoints are stored in Slate and include:
- What was accomplished (`done`)
- What should happen next (`next`)
- Current blockers (`blockers`)
- Key decisions made (`decisions`)
- Files touched (`files`)

### Resumption

When `roland work` is called, it shows a briefing:

```
>>> Resuming task st-a1b2
    Workspace: /home/user/.roland/tasks/st-a1b2-fix-auth

Latest Checkpoint:
  Done:      Implemented auth middleware
  Next:      Add unit tests
  Blockers:  Need API key
  Time:      2025-01-15 14:30

  backend:  dirty (2 changed)
  frontend: clean
```

This gives the agent (and the human) full context for seamless continuation.

### Task context hook

The `roland-task-context` hook (installed in task directories) automatically injects the current task details and latest checkpoint at the start of every agent session. This means even if the agent is restarted, it has context.
